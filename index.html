Architecture v1

System overview

A new standalone service (dextr-marketing-svc) owns hotels, campaigns, email delivery, and analytics. deepagent handles website scrape, KB generation, and filling template JSON properties from hotel knowledge + campaign details.

End-to-end: Create hotel → scrape (profile + KB + assets) → sign in → pick layout → create campaign row → deepagent fills JSON → preview & edit → recipients → send or schedule → analytics.

What deepagent does

deepagent does not produce free-form “campaign copy” as the main deliverable. The campaign row is created in marketing-svc. deepagent fills the properties in the template JSON schema.

marketing-svc creates campaign draft — title, description, template_id, hotel_id
Inputs to deepagent — hotel KB (markdown URL), campaign details, template json_schema, classified assets
deepagent output — values for each JSON slot: headlines, bodies, image URLs, CTA, subject, preheader
marketing-svc saves campaign_json → renderer merges template HTML + JSON → html_url

  • Build KB Markdown file
  • Upload assets to S3
  • Call onboard APIs
  • Fill template JSON properties from KB + campaign context
  • Prompt-edit JSON on user request

Flexible data — metadata jsonb

Hotel and campaign data keeps changing. Instead of adding a new column for every field, store evolving data in a metadata jsonb column.

TableStable columnsmetadata jsonb holds
hotelsname, website_url, timezone, onboard_statusExternal ids, feature flags, custom fields — link to central auth later
hotel_profilessender_from_email, sender_from_nameEntire scraped profile blob: amenities, dining, rooms, brand
campaignshotel_id, template_id, status, scheduled_atExtra campaign fields without migrations
email_templateskey, html_url, json_schema, promptLayout variants, future A/B config
usersusername, role, hotel_idPreferences, audit extras
Trade-off: jsonb is flexible and avoids constant schema migrations. Promote a field to a real column only when it is stable and needs indexing. For v1 this is the right default — requirements will change.

Step 1 — Onboarding entry point

Separate tenant table (hotels) and user table (users) in the marketing service, with dedicated onboarding APIs.

TablePurposeKey columns
hotels Marketing tenant name, website_url, timezone, onboard_status, metadata jsonb
users Login (Option B) username, password_hash, role, hotel_id
MethodPathPurpose
POST/api/v1/hotelsCreate hotel entry
GET/api/v1/hotels/:idRead status
POST/api/v1/usersCreate scoped user
Product linking to external systems is not required at onboard. When central auth is integrated, store external ids in hotels.metadata — no dedicated columns needed now.

Step 2 — Onboarding flow

Required fields → deepagent scrapes → saves profile, KB, assets → onboard_status = ready.

Required fields

  • name, website_url, timezone
scraping — deepagent crawls website
profile — PUT /hotels/:id/profile (stored in hotel_profiles.metadata)
KB — knowledge.md on S3 → POST /hotels/:id/kb { kb_url, version }
assets — POST /hotels/:id/assets { url, classification }
ready — profile + KB + min assets gate passes
KB format: Markdown (knowledge.md) — agent-readable sections. Format defined in deepagent skill.

Step 3 — Sign in

Two authentication options. Both are supported; central auth is preferred.

Preferred

Option A — Central auth

  • User signs in via central auth / SSO
  • JWT passed to marketing-svc
  • Service validates token and maps user → hotel
  • Dashboard embedded or standalone URL

Option B — Marketing login API

  • POST /api/v1/auth/login
  • Username + password against users table
  • Returns marketing JWT (hotel_id + role)
  • Roles: SUPERADMIN, HOTEL_ADMIN, HOTEL_USER

Step 4 — Layout templates

Predefined HTML layouts. Each has HTML (S3), JSON schema (DB), and prompt (DB).

PieceStoredPurpose
HTML layoutS3 → html_urlFixed email structure
JSON schemaDB json_schemaProperties deepagent fills
PromptDB promptLLM instructions per layout

Step 5 — Campaign creation

Campaign row lives in marketing-svc. deepagent fills JSON. Renderer produces HTML.

POST /campaigns — draft: hotel_id, template_id, title, description
POST /campaigns/:id/generate — deepagent fills campaign_json
Renderer — template HTML + campaign_json → html_url

Step 6 — Frontend preview & editor

  • GET /campaigns/:id — html_url, campaign_json, subject
  • Visual edit → PUT /campaigns/:id
  • Prompt edit → deepagent updates JSON → re-render

Step 7 — Recipients

Single email or CSV upload → campaign_recipients

Step 8 — Send email

Load html_url → inject tracking → SES → update status to sent.

Step 9 — Scheduling

scheduled_at + cron worker → same send path as Step 8.

Step 10 — Analytics

Open pixel + click redirect → tracking_events → GET /campaigns/:id/stats
